# LN Advisor

LN Advisor is a Lightning node analysis and recommendation app.

The app runs in the browser, connects to the node through LNC, prepares the request locally, and can send a reduced inference payload to a verified service running on Phala.

This repository currently uses the Props pipeline pattern for inference in two user-facing workflows:

1. `Channels`
2. `Opening Recommendations`

## What "Props pipeline for inference" means here

In this codebase, the Props-style inference flow is:

1. read sensitive node data locally in the browser
2. normalize that data into a stable internal node-state shape
3. apply a privacy policy to produce a reduced payload
4. let the operator inspect the exact outgoing request
5. send that reduced payload to the verified service
6. receive a recommendation, signed result bundle, and verification data
7. verify the returned result in the UI

The important point is that the browser does the initial extraction and reduction step before the verified service sees the request body.

## What runs locally in the browser

The browser is responsible for:

1. connecting to the node through LNC
2. collecting the data needed for the selected workflow
3. building a normalized node-state object
4. applying the privacy policy
5. showing the exact outgoing request in `Review Request` and `Request Inspector`
6. displaying the verification result after the service responds

For the current verified flows, the browser works from local node data such as:

1. channel state
2. fee policy state
3. forwarding history
4. mission control pairs
5. graph data
6. node centrality metrics

That raw local state is not what gets sent to the verified service.

## Normalization and privacy transform

The normalization layer converts the Lightning snapshot into a stable internal structure. That structure includes channel-level state, peer-level aggregates, potential peers, totals, forwarding counts, revenue, failure counts, mission control rates, and centrality values.

The privacy layer then applies one of the supported privacy modes:

1. `full_internal`
2. `feature_only`
3. `banded`

The verified service path uses `feature_only`.

`feature_only` keeps the features needed for inference while removing direct raw identifiers from the request body. Instead of sending the original channel ids and peer pubkeys, the payload uses deterministic local references such as `channel_0001` and `peer_0001`. It also sends feature values such as balance ratios, fee rates, forwarding counts, revenue, failure counts, centrality, and mission-control-derived rates.

For opening recommendations, the same pattern is used for `potentialPeers`.

## What is sent to the verified service

The verified request body is a reduced telemetry object. The exact body is visible in the UI before sending.

The browser sends the reduced payload to:

1. `POST /api/recommend?full=true`

The verified run then gathers the service evidence needed for verification:

1. `GET /health`
2. `GET /info?full=true`
3. `GET /attestation?full=true`
4. `POST /api/verify`

In development, the browser uses the Vite proxy path:

1. `/__phala/...`

In production, the browser talks to the verified service endpoint directly through the configured app URL built into the frontend.

## What is running on Phala

The verified service is the minimal Phala deployment under:

1. `deploy/phala-minimal-prototype/`

It exposes these endpoints:

1. `GET /health`
2. `GET /info`
3. `GET /attestation`
4. `POST /api/recommend`
5. `POST /api/verify`

For the verified path used by the UI, the important operations are `recommend` and `verify`.

The service currently performs deterministic recommendation logic over the reduced payload. The current model identifier embedded in the result is:

1. `prototype-fee-forward-v1`

That model version is included in the signed result bundle. The result digest is computed over:

1. `arbVersion`
2. `provider`
3. `modelVersion`
4. `privacyMode`
5. `bindingMode`
6. `issuedAt`
7. `inputHash`
8. `outputHash`

This means the returned result is bound to both the transformed input and the exact output generated by the service.

## What the service returns

A verified run returns:

1. a transformed snapshot
2. a recommendation set
3. an ARB-style signed result bundle
4. a source receipt
5. live app info
6. live app attestation
7. a verification result

The recommendation set currently contains:

1. fee recommendations for channel analysis
2. opening recommendations for candidate peer selection
3. summary counts

The ARB bundle currently includes:

1. `modelVersion`
2. `privacyMode`
3. `bindingMode`
4. `inputHash`
5. `outputHash`
6. `digest`
7. `signature`
8. `sourceReceiptHash`
9. `attestation`

## Signer and signer policy

The current verified deployment uses the released signer path rather than the older plain env-HMAC path.

The signer runtime reported by the verified service includes:

1. signer provider id
2. signer provider type
3. signing mode
4. key source
5. key provider id
6. released signer policy

The released signer mode in the Phala minimal service is:

1. `released_keyring`

The signing mode reported in verified runs is:

1. `released_signer`

The released signer policy includes:

1. required execution mode
2. allowed provider ids
3. allowed measurements
4. allowed quote formats
5. key id

This means the signing path is gated by attestation policy before the service signs the result digest.

## TEE runtime attestation

The Phala deployment uses runtime attestation from the dstack environment.

In the current setup:

1. the service reads runtime information from the mounted dstack runtime
2. the service exposes that runtime state through `/info` and `/attestation`
3. the ARB attestation evidence includes the runtime quote and related metadata

The service reports the attestation source in `/health` and in the UI. The current runtime source used by the verified path is:

1. `dstack_runtime`

The attestation evidence included with the result contains fields such as:

1. provider id
2. execution mode
3. quote format
4. quote
5. quote hash
6. measurement
7. report data digest
8. app compose hash

The report data digest is bound to the ARB digest. That gives the service a link between the returned recommendation bundle and the runtime evidence returned by the service.

## Live app verification

The verified flow does not stop at checking a local signature.

The UI also asks the service to verify the live app evidence by combining:

1. the returned ARB bundle
2. the service's `/info` response
3. the service's `/attestation` response
4. the source receipt

The verify path checks:

1. transformed input hash
2. recommendation output hash
3. ARB digest
4. signer metadata
5. signature validity
6. source receipt hash
7. attestation policy
8. live app evidence policy

The live verification path also supports runtime quote verification through Phala's cloud attestation API.

The verification response includes:

1. `cloudVerification`
2. `signerPolicy`
3. `attestationPolicy`
4. `sourceReceiptPolicy`
5. `liveAppEvidencePolicy`

## Measurement pinning

The verified deployment is configured with a pinned runtime measurement.

This is visible in:

1. `GET /health`
2. the UI `Verification Status` panel

The measurement policy reports:

1. `allowedMeasurement`
2. whether it is pinned
3. whether a placeholder is still in use
4. whether pinning is still recommended

The verified path is considered correctly pinned when:

1. `pinned` is `true`
2. `placeholderDetected` is `false`

This is the control that binds verification to the expected runtime measurement rather than accepting a placeholder.

## What "pinned model" means in the current implementation

The current implementation has two distinct pinned concepts.

The first is the runtime measurement pin described above.

The second is the model identifier inside the signed result bundle. The service currently emits a fixed model version string:

1. `prototype-fee-forward-v1`

That model version is part of the signed digest input. So the returned ARB is not only tied to the output values, it is also tied to the model version declared by the service for that run.

This is not a separate model registry yet. It is a fixed model identifier embedded into the signed bundle and verified through digest binding.

## What the UI shows after a verified run

The current verified pages show three main result areas.

The first is the recommendation itself.

For `Channels`, that includes:

1. suggested action
2. target fee rate
3. confidence
4. reasons
5. selected signals

For `Opening Recommendations`, that includes:

1. candidate peers
2. scores
3. reasons
4. key signals such as centrality, reliability, and capacity bands

The second is `Verification Status`. That panel shows:

1. signer
2. signer type
3. runtime check
4. measurement
5. runtime source
6. live verification

The third is `Request Inspector`. That panel shows:

1. raw local counts used for the run
2. normalized intermediate state
3. final outgoing payload
4. exact network requests

## Review before send

Verified flows use an explicit review step.

On the first verified run in a page session:

1. the primary action opens `Review Request`
2. the modal shows the exact request body and request plan
3. `Send Request` starts the verified run

After one successful reviewed run:

1. the primary action becomes `Run Verified`
2. a secondary `Review Request` button remains available

This lets the operator inspect the payload first, then repeat the same verified workflow with less friction.

## Current verified workflows

### Channels

The `Channels` page uses the reduced single-channel payload to generate fee policy recommendations.

The verified run returns:

1. a fee recommendation
2. verification data
3. runtime evidence

### Opening Recommendations

The `Opening Recommendations` page uses reduced graph and node-state features to rank candidate peers.

The verified run returns:

1. candidate peer recommendations
2. verification data
3. runtime evidence

## Local setup

Install dependencies:

```bash
pnpm install
```

Create `.env` from `.env.example`:

```bash
cp .env.example .env
```

Minimum values:

```env
VITE_ENABLE_PHALA_VERIFIED_UI=true
VITE_API_BASE_URL=http://127.0.0.1:8787
API_PORT=8787
```

Start the frontend:

```bash
pnpm dev --host
```

Open:

```text
http://localhost:5173
```

Start the local API only if you need the standard local routes:

```bash
pnpm api
```
